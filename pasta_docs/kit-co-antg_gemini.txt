# CLAUDE.md — Regras de Vibe Coding> Leia este arquivo antes de iniciar qualquer sessão de IA.> Referencie-o no seu primeiro prompt: "Leia o CLAUDE.md e respeite todas as regras."---## Regra 1 — Arquitetura Modular (Arquivos < 600 linhas)

A IA trabalha com context window. Arquivos grandes causam:- Perda de contexto- Alucinações- Regressões em código que já funcionava**Ação:** Dividir código em arquivos menores, com responsabilidade única.

Exemplos:- `AuthService.js`- `UserRepository.js`- `LoginPage.jsx`- `utils/validation.js`**Por quê?** Quando você coloca tudo em um arquivo gigante, a IA perde o contexto das partes que já funcionam e começa a quebrar coisas. Arquivos pequenos (< 600 linhas) permitem que a IA entenda melhor o que está fazendo.---## Regra 2 — Conhecer os Building Blocks Antes de Prompar**A IA não protege de problemas que o desenvolvedor não conhece.**> "Seu trabalho não é escrever cada linha de código. É saber o que pedir."### Conceitos que devem ser mencionados nos prompts quando relevantes:- **Race conditions** — múltiplos usuários editando ao mesmo tempo- **Optimistic locking / Pessimistic locking** — como lidar com conflitos- **Idempotência** — operações que podem ser repetidas com segurança- **Consistência eventual** — dados distribuídos- **Autenticação vs Autorização** — quem é você vs o que você pode fazer- **Rate limiting** — proteção contra abuso de APIs- **Escalabilidade** — o código funciona com 1 usuário? E com 1 milhão?### Exemplo de prompt BOM:

Crie um sistema onde múltiplos usuários podem editar o mesmo documento. Implemente optimistic locking para evitar race conditions. Use timestamps para detectar conflitos. Se houver conflito, o último a salvar vence (last-write-wins).

### Exemplo de prompt RUIM:

Crie um editor de documentos colaborativo.

## Regra 3 — Uma Task por Sessão → Testar → Commitar

Escolher UMA task do plan.md

Abrir nova sessão de IA

Executar a task

Testar manualmente

Se OK → git add . && git commit -m "feat: "

Só então passar para a próxima task

### Por quê?



Com modelos menores/gratuitos, o risco de o modelo quebrar algo que já funcionava é real. Commits frequentes criam checkpoints que permitem rollback instantâneo.



### Fluxo de commit recomendado:bash

Copiar

# Após testar a task manualmente e tudo estar OK:git add .git commit -m "feat: descrição clara da task"git push origin main# Próxima task









### Mensagens de commit padrão:



- `feat: nova funcionalidade`

- `fix: correção de bug`

- `refactor: reorganização de código`

- `chore: setup, dependências, etc`



---



## Resumo das 3 Regras



| Regra | O quê | Por quê |

|---|---|---|

| **Modular** | Arquivos < 600 linhas | IA entende melhor, menos alucinações |

| **Informado** | Mencione building blocks nos prompts | IA não protege de problemas desconhecidos |

| **Incremental** | Uma task, testar, commitar | Checkpoints seguros, fácil rollback |

Aqui está o guia para sua decisão:

1. Escolha "Allow for Workspace" se:
Você está trabalhando em um projeto específico agora e quer que o Gemini 3 Flash tenha acesso apenas aos arquivos dessa pasta (d:\__IDEs_projetos_ia).

Vantagem: É a opção mais segura. Garante que a IA não saia "bisbilhotando" outras pastas do seu computador que não têm relação com o código atual.

Uso ideal: Para sessões de Vibe Coding onde você segue a Regra 3 (Uma Task por Sessão) do seu CLAUDE.md.

2. Escolha "Allow Globally" se:
Você usa essa mesma pasta raiz para todos os seus projetos e não quer que o Antigravity peça permissão toda vez que você abrir um novo repositório dentro dela.

Vantagem: Maior agilidade e menos interrupções no fluxo de trabalho.

Risco: Se você tiver arquivos sensíveis (como chaves de API esquecidas ou documentos pessoais) em outras pastas, a IA poderá lê-los se o prompt a direcionar para lá.

🚀 Veredito do MEU PM:

Clique na seta ao lado de "Allow" e selecione "Allow for Workspace".

O Spec-Driven Development (SDD), conforme ilustrado no diagrama da imagem enviada e detalhado nas práticas de engenharia agentic, representa a evolução do "vibe coding" para o desenvolvimento estruturado com IAs. Em vez de pedir que a IA escreva código diretamente a partir de uma ideia vaga, o SDD insere camadas de planejamento e validação que garantem previsibilidade e reduzem erros.

Com base no diagrama e no seu projeto CLI Kit CO, aqui está a análise detalhada de como implementar esse fluxo:



1. Análise do Fluxo SDD (Baseado na Imagem)

O diagrama define um ciclo de vida linear e rigoroso:



IDEIA + PRD: O ponto de partida onde você define a "vibe" e os objetivos de negócio. No seu caso, este é o documento PRD.md que já criamos.

SPEC (Specification): Esta é a fase crítica que falta no vibe coding tradicional. Em vez de pular para o código, a IA gera um blueprint técnico detalhado (ex: implementation_plan.md).

IMP (Implementation): A execução do código baseada estritamente na especificação aprovada.

TEST: Validação automática (unitária, integração ou UI via Playwright) para garantir que o código atende à SPEC.

DEPLOY: A entrega final após a verificação de sucesso.

2. Implementação no CLI Kit CO / Antigravity

Para tornar o seu Antigravity "mais inteligente" e auto-modificável usando SDD, você deve configurar os seguintes componentes técnicos:



Modo de Planejamento (Plan Mode): Utilize o modo /plan (ou Shift+Tab) do Antigravity para forçar a criação do artefato de SPEC antes de qualquer alteração de arquivo.

Constituição do Projeto (constitution.md): Defina regras fundamentais em um arquivo .specify/memory/constitution.md. Este arquivo serve como a "lei" que a IA deve consultar para garantir que a SPEC e a IMP sejam consistentes com os padrões da sua organização.



Comandos de SDD no Terminal:

/specify: Cria a especificação técnica em uma nova branch.

/clarify: Permite que o agente faça perguntas sobre áreas subespecificadas antes de planejar.



/analyze: O agente revisa o impacto das mudanças no repositório inteiro antes de implementar.



3. Tornando o Sistema Auto-Modificável

Para que o seu kit se modifique e aprenda (conforme discutimos sobre o padrão Darwin-Gödel), o SDD deve ser recursivo:



Feedback de Teste para SPEC: Se o passo TEST falhar, o agente não deve apenas tentar "corrigir o código", ele deve voltar ao passo SPEC, atualizar o plano técnico e só então re-implementar.

Artifact-Based Transparency: Use os artefatos gerados (como walkthrough.md) como memória de longo prazo. O agente deve ler esses documentos de missões passadas para não repetir erros arquiteturais em novas especificações.

4. Orquestração de Modelos no SDD

Uma prática recomendada para economizar e aumentar a inteligência é dividir os modelos por fase :



Fase SPEC: Use modelos de alto raciocínio (Claude 3.5 Sonnet ou Gemini 3 Pro) para garantir que a arquitetura seja sólida.

Fase IMP: Use modelos mais baratos e rápidos (Qwen 2.5 Coder ou Big Pickle) para a escrita mecânica do código, já que o "pensamento" foi resolvido na especificação.

Ao adotar o SDD, você transforma o CLI Kit CO de uma ferramenta de "chutar código" em uma fábrica de software agentic, onde cada commit é verificado contra uma especificação técnica formalmente aprovada.
